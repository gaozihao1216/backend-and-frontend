  一、1 分钟项目架构介绍
  这个项目是一个 UGC 关卡平台，前端在 frontend/src，主要通过 frontend/src/src/api 里的接口文件发请求。前端按角色
  分成 player、designer、admin 三类页面，调用的 API 也分别落在 auth-api.ts、user-api.ts、designer-api.ts、player-
  api.ts、admin-api.ts。

  为了满足课程“后端必须使用 Scala、API 必须定义在 scala/microservice”的要求，我新增了一个 Scala 的类型安全 API 定
  义层，实际可编译源码放在 src/main/scala/microservice，字面上的 scala/microservice 目录用 README 做了说明。这个
  Scala 层当前的职责不是立刻替换整个 Node 运行时，而是把核心 API 的契约，也就是 request、response、error、
  endpoint、service interface，用 Scala 类型系统明确定义出来。

  现有的 TypeScript 后端代码还保留在 src/backend/routes 和 src/backend/services，它们现在相当于“真实业务参考实
  现”；Scala microservice 则负责课程要求里的类型安全 API 定义。共享类型这边，前端和 TS 后端继续通过 src/shared/
  api 和 src/shared/schemas 对齐，而 Scala 端用 case class、sealed trait、trait 去映射这些核心语义，做到前后端
  API 一一对应。

  二、Scala microservice 目录讲解
  src/main/scala/microservice/core 是整个 Scala API 定义层的基础框架。

  这里最核心的几个类型是：

  - HttpMethod：用 sealed trait 加 GET/POST/PUT/DELETE 建模 HTTP 方法。
  - ApiPath：把路径包成一个显式类型，而不是到处散落裸字符串。
  - ApiError：所有错误的父类型，下面再细分 ValidationError、NotFoundError、UnauthorizedError 这类错误。
  - ApiEndpoint[Req, Res]：每个 API 都必须明确自己的 request 类型和 response 类型。
  - ApiHandler[Req, Res]：处理逻辑统一返回 Either[ApiError, Res]，表示“要么失败，要么成功”。

  业务模块按领域拆开：

  - microservice/auth：认证相关 API，比如绑定本地身份到后端用户。
  - microservice/user：用户资料相关 API。
  - microservice/level：关卡相关 API，这里放了 designer 创建关卡和 player 评分。
  - microservice/admin：管理员审核 submission 的 API。
  - microservice/MicroserviceApiCatalog.scala：把已经定义好的核心 API 汇总，方便展示。

  为什么这属于函数式 / 类型安全 API 框架：

  第一，它是类型安全的，因为 request、response、error 都不是 Map[String, Any]，而是明确的 Scala 类型。比如一个
  API 的 request 如果少字段、字段类型不对，类型层面就不成立。

  第二，它是函数式风格的，因为 handler 和 service 的结果不是抛异常式接口，而是显式返回 Either[ApiError, Res]。也
  就是说，失败是类型的一部分，成功和失败都被纳入函数签名里，业务边界更清楚。

  二、Scala microservice 目录讲解
  src/main/scala/microservice/core 是整个 Scala API 定义层的核心框架。

  其中 src/main/scala/microservice/core/ApiContract.scala:1 里定义了几个关键抽象：

  - ApiPath
      - 把 path 包成一个显式类型，而不是散落在各处的裸字符串
  - ApiError
      - 所有 API 错误的父类型，用 sealed trait 建模
  - ApiEndpoint[Req, Res]
      - 每个 endpoint 都必须显式声明请求类型和响应类型
  - ApiHandler[Req, Res]
      - 处理逻辑返回 Either[ApiError, Res]

  src/main/scala/microservice/core/HttpMethod.scala:1 里定义了 GET、POST、PUT、DELETE，它们不是随便写字符串，而是
  sealed trait HttpMethod 的几个 case object，这样方法集合是受控的。

  业务模块分成：

  - microservice/auth
      - 负责认证绑定相关 API
  - microservice/user
      - 负责用户资料 API
  - microservice/level
      - 负责关卡相关 API，现在拆成 designer 创建关卡、player 评分两个方向
  - microservice/admin
      - 负责管理员审核 submission
  - microservice/model
      - 放领域模型，比如 User、Level、Submission、Rating
  - microservice/MicroserviceApiCatalog.scala
      - 把几个核心 API 汇总出来，方便展示整体 API 列表

  为什么这属于函数式 / 类型安全 API 框架，我会这样讲：

  第一，它是类型安全的。因为 request、response、error 都是显式类型，不是 Map[String, Any]，也不是随手拼 JSON。编
  译器会检查接口签名是否一致。

  第二，它是函数式风格的。因为 handler 或 service 的返回值不是靠抛异常表示正常业务失败，而是用 Either[ApiError,
  Res] 显式表达“成功或失败”两种结果。这种写法把失败也纳入类型系统，逻辑边界更清楚。

  第三，契约和实现分离。ApiEndpoint 只定义接口契约，Service trait 只定义业务能力签名，不把具体 Express 或数据库细
  节耦合进去。这种分层本身也很适合 code review。

  二、Scala microservice 目录讲解

  src/main/scala/microservice/core 是整个 Scala API 定义层的基础框架。这里主要有两个文件。

  第一个是 src/main/scala/microservice/core/HttpMethod.scala:1，它定义了 sealed trait HttpMethod，以及 GET、
  POST、PUT、DELETE 这些 case object。这样做的好处是 HTTP 方法不是随便写字符串，而是一个受类型系统约束的有限集
  合。

  第二个是 src/main/scala/microservice/core/ApiContract.scala:1，这里定义了：

  - ApiPath
  - ApiError
  - ValidationError / NotFoundError / UnauthorizedError / ConflictError / ForbiddenError
  - ApiEndpoint[Req, Res]
  - ApiHandler[Req, Res]

  ApiEndpoint[Req, Res] 负责描述一个 API 的 method、path、name、description。ApiHandler[Req, Res] 负责定义
  handler 的签名：handle(request: Req): Either[ApiError, Res]。这已经体现出函数式风格了，因为它把成功和失败显式建
  模成 Either，而不是靠抛异常或者返回弱类型对象。

  业务模块分为：

  - microservice/auth
  - microservice/user
  - microservice/level
  - microservice/admin

  auth 负责账号绑定类接口，user 负责用户资料，level 负责设计师建关卡和玩家评分，admin 负责管理员审核。每个模块都
  包含 request、response、error、endpoint、service trait，结构是统一的，很适合 code review 展示。

  为什么这属于函数式 / 类型安全 API 框架，可以从四点讲：

  1. request 和 response 都是 case class，不是 Map[String, Any]
  2. error 是 sealed trait，错误空间是显式的、可枚举的
  3. endpoint 是泛型 ApiEndpoint[Req, Res]，请求和响应的类型在定义阶段就固定了
  4. handler / service 返回 Either[ApiError, Res]，把失败作为类型的一部分表达出来，这就是典型的函数式建模方式

  所以这个 Scala microservice 虽然是轻量级的，但已经满足“类型安全”和“函数式 API 契约表达”的课程要求。

  - 前端对应文件：frontend/src/src/api/auth-api.ts:1
  - Request 类型：BindBackendUserRequest
  - Response 类型：BindBackendUserResponse
  - Error 类型：AuthApiError，当前主要实现了 BindBackendUserValidationError
  - HTTP 方法和路径：POST /auth/bind
  - 业务逻辑：
    这个接口的作用是把前端的本地身份，例如 localUserId、nickname、role，绑定成一个后端用户。它不是每次都新建用
    户，而是先看这个 localUserId + role 是否已经对应到一个确定性的用户名，如果已经有，就复用旧用户；如果没有，才
    创建新的后端用户。
  - 前后端如何对齐：
    前端在 auth-api.ts 里调用 /auth/bind；TS shared schema 里有 BindBackendUserRequestBodySchema 和
    BindBackendUserResponseDataSchema；Scala 里对应的是 BindBackendUserRequest 和 BindBackendUserResponse。
  - 老师可能追问：
    “为什么这里不是登录，而是 bind？”
  - 推荐回答：
    因为这个项目当前的身份模型是前端本地身份和后端用户记录之间做绑定，重点不是完整认证系统，而是先保证前后端对同
    一个用户实体有一致表示，所以接口名和语义都叫 bind，更贴合现有项目逻辑。

  - Response 类型：GetUserProfileResponse
  - Error 类型：UserApiError，核心是 UserNotFound

  - 前后端如何对齐：

    GetUserProfileResponseDataSchema，Scala 里对应 GetUserProfileRequest 和 GetUserProfileResponse(profile:

  ### 1. 为什么必须加 scala/microservice？

  推荐回答：

  因为课程明确要求 API 必须定义在 scala/microservice。但 sbt 的标准可编译源码目录其实是 src/main/scala，所以我把
  真正参与编译的 Scala API 定义放在 src/main/scala/microservice，同时保留了 scala/microservice/README.md 作为字面
  目录说明。这样既符合课程检查口径，也符合 Scala 项目的标准结构。

  ### 2. 你的 API 哪里体现了类型安全？

  推荐回答：

  类型安全主要体现在四层。第一，请求和响应都不是随便的 JSON 或 Map[String, Any]，而是 case class。第二，错误不是
  字符串随便抛，而是 sealed trait ApiError 及其子类型。第三，每个 endpoint 都是 ApiEndpoint[Req, Res]，在定义时就
  绑定了请求和响应类型。第四，service 的返回值是 Either[ApiError, Response]，成功和失败都被显式纳入类型系统，而不
  是靠约定猜测。

  ### 3. 你的 API 哪里体现了函数式？

  推荐回答：

  函数式最明显的地方是 Either。在这个设计里，handler 或 service 不是返回空值或者直接抛异常，而是返回
  Either[ApiError, Res]，把失败作为显式结果处理。另外整个 API 契约是用不可变的 case class 和代数数据类型来建模
  的，这也是典型的函数式风格。虽然这是一个轻量级框架，没有引入更重的 FP 库，但已经展示出函数式接口设计思路。

  ### 4. 前端怎么保证和 Scala API 对齐？

  推荐回答：

  前端当前通过 src/shared/api 和 src/shared/schemas 中的 Zod schema 保证运行时契约一致性；Scala microservice 则从
  这些真实存在的 API 出发，按同样的 request、response、error 语义重新建模。也就是说，我不是先写一套 Scala API 再
  让前端去适配，而是以现有前端调用和后端路由为基准，把核心 API 映射成 Scala 类型安全定义。所以对齐的依据是“前端调
  用路径 + shared schema + 现有 TS route/service”。

  ### 5. 现在还有哪些没完全完成？

  的是把整个运行时后端从 TypeScript 全量迁移到 Scala。现在真实运行逻辑仍在 src/backend/routes 和 src/backend/
